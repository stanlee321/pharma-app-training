# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**TivaTrainer** — a SwiftUI (iOS 17+) pharmacokinetic simulation app for anesthesiologists. It computes and visualizes drug concentration curves for Total Intravenous Anesthesia (TIVA) using Target-Controlled Infusion (TCI). An anesthesiologist picks a drug/PK model, enters patient biometrics, sets target concentrations, and the app returns bolus/infusion dosing plus plasma (Cp) and effect-site (Ce) curves, rendered as an interactive chart and a 2D/3D compartmental animation.

Four PK models are implemented: Propofol (Marsh, Schnider), Remifentanil (Minto), Dexmedetomidine (Hannivoort) — all as a 3-compartment mammillary model + effect site (4-state ODE).

## Build & run

The Xcode project is **generated by [XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml`** — do not hand-edit `PharmaApp.xcodeproj`, it is a build artifact. Sources are picked up automatically from the `PharmaApp/` directory tree, so:

```bash
xcodegen generate          # regenerate after ANY file add/rename/move — new .swift files won't be in the target otherwise
open PharmaApp.xcodeproj    # then select a team in Signing, pick a device/simulator, Cmd+R
```

Command-line build/test (no XCUITest scheme is configured; adjust the destination as needed):

```bash
xcodebuild -project PharmaApp.xcodeproj -scheme PharmaApp \
  -destination 'platform=iOS Simulator,name=iPhone 15' build
```

There is **no test target and no lint config** in this repo — the "Validation Reference" (`docs/rust-engine-spec/04_Validation_Reference.md`) holds hand-calculated expected values but they are not wired into an XCTest suite. Do not claim tests pass; there are none to run.

### Regenerating mock simulation data

The app bundles pre-computed simulation snapshots in `PharmaApp/Resources/MockData/*.json`. Regenerate them with the **standalone script** (its PK types are duplicated inline so it runs without the app):

```bash
swift generate_mock_data.swift   # writes 5 JSON files into PharmaApp/Resources/MockData/
```

Note: `PharmaApp/Engine/MockDataGenerator.swift` is an *in-app* equivalent (uses the real engine types), separate from the standalone script. Keep the two scenario lists in sync if you change either.

### TestFlight

Bump `CURRENT_PROJECT_VERSION` in `project.yml` for each upload, then `xcodegen generate`, then Xcode → Product → Archive → Distribute.

## Architecture

### The engine abstraction is the central seam

All PK/PD math lives behind `PKEngineProtocol` (`Engine/PKEngineProtocol.swift`), with two implementations selected by a single compile-time switch:

```
PKEngineProtocol
 ├── MockPKEngine   (Engine/MockPKEngine.swift)  — the ONLY working engine today; a real RK4 ODE solver
 └── RustPKEngine   (Engine/RustPKEngine.swift)  — production stub; every method is fatalError()
```

`EngineProvider.useMock` (`Engine/EngineProvider.swift`) chooses between them. It is `true` and must stay true until the Rust C-FFI library exists — flipping it to `false` today crashes the app. The Rust replacement is fully specified in `docs/rust-engine-spec/` and `docs/technical/02_FFI_Integration_Guide.md`; `SimulationTypes.swift` deliberately mirrors the `#[repr(C)]` FFI contract so the Swift call sites don't change when the engine is swapped.

`MockPKEngine` is **not fake data** — it integrates the 4-state ODE with RK4 and runs the BET (Bolus-Elimination-Transfer) TCI algorithm. Per-model patient parameters (V1/V2/V3, clearances, ke0) are computed from covariates in the `*Params(patient:)` methods.

### Unit conventions (critical — the most common source of bugs)

The engine works internally in **mg** (amounts) and **mg/ml** (concentrations) and **ml** (volumes). Display units differ per drug and are reached by multiplying by `PKParams.displayFactor`:
- Propofol → `×1000` (mg/ml → mcg/ml)
- Remifentanil / Dexmedetomidine → `×1_000_000` (mg/ml → ng/ml)

Rate constants (k10, k12, …, ke0) are **per minute**, but time in the simulation loop is tracked in **seconds** — the RK4 step is called with `dt/60` to reconcile. Target concentrations coming from the UI are in display units and must be divided by `displayFactor` before entering the engine. When touching any math, check which unit space you are in.

### State & data flow

- One `AppState` (`App/AppState.swift`, `@MainActor @Observable`) is the single source of truth: selected drug, `PatientProfile`, `[TargetEvent]`, `SimulationOutput`, playback state, and the engine handle. It is injected once via `.environment(appState)` in `PharmaApp.swift`.
- ViewModels (`SimulationViewModel`, `CompartmentalViewModel`) are thin `@Observable` wrappers that hold a reference to `AppState` and derive chart/animation state from `appState.simulationOutput`. They do not own the data.
- Screen flow: `RootView` hosts a `NavigationStack`; navigation is driven by the `Route` enum (`App/Route.swift`) through a `NavigationPath` binding that is threaded down into each view. Order: Drug Selection → Patient Input → Simulation (with an embedded segmented Graph/Compartments switch). The launch modal is a one-time overlay gated on the `hasSeenAnnouncement_v1` `@AppStorage` flag.
- Drug database: `DrugModel`/`DrugDatabase` load `Resources/drug_models.json` with `.convertFromSnakeCase` decoding, falling back to `DrugDatabase.hardcoded` if the bundle load fails. Each `DrugModel.rustModelId` (`"marsh"`, `"schnider"`, `"minto"`, `"hannivoort"`) is the key that maps a UI drug entry to an engine model — this string is what gets passed to the engine, not the `DrugModel.id`.

### Two interaction hotpaths

- **Target drag** (`SimulationViewModel`): dragging the target node calls `engine.computeTCI(...)` on every change to update bolus/infusion readouts live — required to stay under 16ms (60fps). Releasing ("Done") triggers a full `engine.simulate(...)` re-run and re-scales the axes.
- **Rendering**: the chart (`Views/Simulation/ChartCanvas.swift`) and the 2D compartmental view (`Views/Compartmental/CompartmentCanvas.swift`) are hand-drawn with SwiftUI `Canvas`. The 3D compartmental view (`Views/Compartmental/CompartmentSceneView.swift`) is SceneKit (`SCNCylinder` glass compartments, `SCNParticleSystem` flow along pipes), toggled via `CompartmentalViewModel.use3D`.

## Conventions to follow

- **Strict concurrency is on** (`SWIFT_STRICT_CONCURRENCY: complete` in `project.yml`). All model/output types are `Sendable`; engines are `@unchecked Sendable`; UI state classes are `@MainActor`. New engine-crossing types must be `Sendable`, and new UI state must respect the main-actor isolation.
- Use the `@Observable` macro (not `ObservableObject`/`@Published`) and `.environment(...)` injection — this codebase is on the iOS 17 observation model throughout.
- Centralized styling lives in `Utilities/DesignSystem.swift` (`AppColors`, `Font` extensions, `Haptics`, `Spacing`). Reuse these rather than introducing ad-hoc colors/fonts; the curve/compartment color meanings (cyan = plasma, green = effect, orange = infusion) are consistent across chart and animation.
