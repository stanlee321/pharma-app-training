# FFI Integration Guide: Rust ↔ Swift

## 1. Overview

The iOS app calls the Rust `pk-engine` library through a C-FFI boundary. Rust compiles to a static library (`libpk_engine.a`) targeting `aarch64-apple-ios`. Swift calls the exported C functions through a bridging header.

## 2. Build Pipeline

### 2.1 Cross-compile Rust for iOS

```bash
# One-time setup
rustup target add aarch64-apple-ios        # Device
rustup target add aarch64-apple-ios-sim     # Simulator (Apple Silicon)

# Build
cd pk-engine
cargo build --release --target aarch64-apple-ios
# Output: target/aarch64-apple-ios/release/libpk_engine.a

# For simulator testing:
cargo build --release --target aarch64-apple-ios-sim
```

### 2.2 Generate C Header

Using `cbindgen`:

```bash
cargo install cbindgen
cbindgen --config cbindgen.toml --crate pk-engine --output include/pk_engine.h
```

`cbindgen.toml`:
```toml
language = "C"
include_guard = "PK_ENGINE_H"
autogen_warning = "/* Auto-generated by cbindgen. Do not edit. */"

[export]
include = ["SimulationInput", "SimulationOutput", "TCIResult", "PatientParams", "TargetEvent", "CurvePoint"]
```

### 2.3 Xcode Integration

1. Add `libpk_engine.a` to Xcode project (link binary)
2. Add `include/pk_engine.h` as bridging header
3. In Build Settings:
   - `Library Search Paths`: path to `.a` file
   - `Objective-C Bridging Header`: `$(PROJECT_DIR)/Engine/pk_engine.h`

## 3. C-FFI Contract

### 3.1 Types Crossing the Boundary

All types must be `#[repr(C)]` in Rust and have C equivalents:

```c
// pk_engine.h (auto-generated)

typedef struct {
    double weight;      // kg
    double height;      // cm
    double age;         // years
    int32_t gender;     // 0 = male, 1 = female
    double dilution;    // mg/ml
} PatientParams;

typedef struct {
    double time;           // seconds from sim start
    double concentration;  // target concentration (ng/ml or mcg/ml)
    int32_t target_type;   // 0 = plasma, 1 = effect-site
} TargetEvent;

typedef struct {
    const char* model_id;       // e.g. "marsh", "schnider"
    PatientParams patient;
    const TargetEvent* targets;
    uint32_t target_count;
    double time_range_seconds;
    double resolution_seconds;  // time step for output points
} SimulationInput;

typedef struct {
    double time;
    double plasma_concentration;
    double effect_concentration;
    double amount_v1;
    double amount_v2;
    double amount_v3;
    double amount_effect;
    double infusion_rate;
} CurvePoint;

typedef struct {
    CurvePoint* points;
    uint32_t point_count;
    double v1_volume;    // liters (for compartment sizing)
    double v2_volume;
    double v3_volume;
    double k10;          // rate constants (for particle animation)
    double k12;
    double k21;
    double k13;
    double k31;
    double ke0;
} SimulationOutput;

typedef struct {
    double bolus_mg;
    double bolus_ml;
    double bolus_mcg_per_kg;
    double infusion_rate_mg_hr;
    double infusion_rate_ml_hr;
    double infusion_rate_mcg_kg_hr;
} TCIResult;
```

### 3.2 Exported Functions

```c
// Full simulation: compute all curves for a set of target events
// Returns 0 on success, negative on error

    const SimulationInput* input,
    SimulationOutput* output

// TCI point query: given current state, what bolus/infusion hits target?
// Used during target node dragging (must be < 5ms)

    const char* model_id,
    const PatientParams* patient,
    const double* current_state,  // [v1, v2, v3, effect] amounts
    uint32_t state_len,
    double target_concentration,
    int32_t target_type,          // 0=plasma, 1=effect-site
    TCIResult* result
);

// Free memory allocated by pk_engine_simulate
void pk_engine_free_output(SimulationOutput* output);

// Get available model IDs (null-terminated array of strings)
const char* const* pk_engine_list_models(uint32_t* count);

// Get validation rules for a model
int32_t pk_engine_get_validation(
    const char* model_id,
    double* min_weight, double* max_weight,
    double* min_height, double* max_height,
    double* min_age, double* max_age
);
```

## 5. Threading Model

```
Main Thread (UI)
  │
  │  User drags target node
  │  ├── Debounce: coalesce rapid drag events (max 60 calls/sec)
  │  └── Dispatch to background:
  │
Background Thread (serial queue)
  │  pk_engine_compute_tci()  ← fast, < 5ms
  │  └── Publish result back to main thread
  │
  ▼
Main Thread updates readouts (Bolus, Infusion values)
```

For full simulation ("Done" pressed):
```
Main Thread → shows loading indicator
  │
Background Thread
  │  pk_engine_simulate()  ← up to 50ms
  │  └── Publish curves back to main thread
  │
Main Thread → updates chart with new curves, hides loading
```

## 6. Memory Safety

- Rust allocates `CurvePoint` array inside `pk_engine_simulate()`
- Swift MUST call `pk_engine_free_output()` after copying data
- Use Swift `defer` to guarantee cleanup:

```swift
var output = SimulationOutput()
defer { pk_engine_free_output(&output) }
let status = pk_engine_simulate(&input, &output)
guard status == 0 else { return nil }
// Copy data from output before defer frees it
```

- String parameters (`model_id`) must be passed as null-terminated C strings
- Use `modelId.withCString { ptr in ... }` in Swift
